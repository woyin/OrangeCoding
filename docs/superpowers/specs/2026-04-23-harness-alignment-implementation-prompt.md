# Harness Engineering 实现代理 Prompt

你现在是 OrangeCoding 项目的实现代理。你的任务不是发散思考，而是**严格、系统、可验证地落地已批准的规格**。本轮必须实现的核心规格文件为：

- `docs/superpowers/specs/2026-04-23-harness-alignment-design.md`

你的总目标是：在不破坏现有 OrangeCoding 架构的前提下，为系统加入一个**独立的 Harness 对齐治理层**，它能够在长时间自治执行中持续锚定原始任务目标，防止代理逐步偏离目标、追逐支线问题，且在重大方向变化时触发明确门控，而不是静默继续执行。

---

## 一、项目上下文

这是一个大型 Rust 工作区项目，当前代码库已经具备：

- `orangecoding-agent`：代理循环、工作流、上下文治理
- `orangecoding-tui`：终端交互界面
- `orangecoding-tools`：工具系统与权限控制
- `orangecoding-config`：配置体系
- `orangecoding-session`：会话与持久化
- `orangecoding-worker` / `orangecoding-control-server`：运行时与控制层

你必须遵守现有项目约束：

1. 优先沿用现有架构与命名习惯，不要随意引入与现有设计冲突的新体系。
2. 错误处理必须使用工作区现有的 `OrangeError` 与 `Result<T>` 约定。
3. 文档与代码注释应保持中文风格，与仓库现有习惯一致。
4. 配置必须接入现有配置体系，而不是额外发明一套平行配置机制。
5. 测试必须符合当前仓库实践；单元测试允许中文测试名，不变式测试遵循现有工作区测试方式。

## 二、你要实现的不是“一个功能点”，而是一条治理边界

请始终记住：本次实现的重点不是把某个 workflow 变得“更聪明”，而是把“目标对齐治理”从工作流内部抽离出来，形成一个清晰、独立、可配置、可观测的 Harness 层。

也就是说：

- 工作流继续决定**如何执行**
- Harness 决定**当前执行是否仍然服务于原始任务**

如果你在实现过程中发现自己把判断逻辑重新塞回 autopilot 或其他 workflow 内部，这通常意味着你正在偏离规格。

---

## 三、必须实现的核心能力

你至少要落地以下能力，除非代码现实约束证明必须拆阶段，但即便拆阶段，也必须保证边界和接口已经正确建立。

### 1. MissionContract

实现一个可持久化的任务契约对象，用来表示长任务在整个生命周期中的“正式任务定义”。它至少应包含：

- 原始任务目标
- 成功标准
- 当前已接受计划摘要
- 禁止偏离的边界或支线类型
- 重大计划变更判定规则
- 审阅门控策略
- Harness 配置快照

它必须能够在中断恢复后重新加载并继续使用，而不是在恢复时重新解释用户目标。

### 2. HarnessSupervisor

实现一个顶层治理入口，负责：

- 初始化 MissionContract
- 控制“有界执行段”
- 在检查点收集 `StepOutcome`
- 调用偏航判定逻辑
- 决定继续执行、触发受控重规划或升级给用户

该组件必须位于 `orangecoding-agent` 的清晰边界中，而不是零散地散布在多个 workflow 文件里。

### 3. StepOutcome / ProgressTracker

工作流在每个执行段结束后，必须能向 Harness 返回结构化结果，而不是只返回模糊文本。这个结构至少要表达：

- 本段完成了什么
- 触及了哪些资源或文件
- 做出了哪些关键决策
- 当前策略的理由
- 当前阻塞点或风险

Harness 应使用这些结构化结果来进行对齐判断和记录。

### 4. DriftDetector

实现一个明确的偏航判定组件。它至少要区分三类结果：

- 仍然对齐，可以继续
- 仍然相关，但触发重大方向变化，需要重规划或审阅
- 已偏离目标，必须阻止自治继续推进

注意：首版最关键的偏航类型，不是“执行失败”，而是**从主目标滑向看起来有价值、实际上不在成功标准里的支线任务**。

### 5. ReplanGate

实现计划变更门控逻辑。重点不是“帮 agent 自动重写计划”，而是**约束哪些变化可以静默继续，哪些变化必须显式升级**。重大计划变更不能绕过门控。

### 6. HarnessConfig

把关键策略做成显式配置项，并接入现有配置系统。首版至少考虑：

- 检查点频率或最大执行段长度
- 重大计划变更阈值
- 偏航检测敏感度
- 置信度下降时的升级策略
- 单次自治执行上限
- 重试 / 重规划 / 请求用户确认的偏好

## 四、实现原则

### 原则 1：分段执行，而不是无限长循环

长任务必须被拆成可检查的“有界执行段”。每个执行段结束后，都必须进入 Harness 检查点，而不是让 workflow 无限自我推进。

### 原则 2：偏航不能静默放行

只要检测到明显偏航，就必须停止自治推进、保留最近已接受状态，并给出具体原因。不要添加“看起来更顺滑”的静默容错逻辑。

### 原则 3：执行域错误与对齐域错误必须分离

- 工具失败、网络异常、模型超时等属于执行域
- 目标偏离、重大计划漂移、无法证明当前动作仍服务于目标属于对齐域

不要把这两类问题混成一个通用 error 分支。

### 原则 4：外层治理、内层复用

优先复用现有 autopilot、boulder 等工作流的执行能力，不要因为要实现 Harness 就推倒已有流程。首版应采取包裹式治理，而不是重写式替换。

### 原则 5：接口先行，内部逐步适配

即便首版不能让所有 workflow 同时完整适配，也必须先把核心接口设计正确：

- workflow 如何产出 `StepOutcome`
- Harness 如何发出继续 / 重规划 / 升级裁决
- MissionContract 如何持久化与恢复

## 五、推荐实施顺序

请按以下顺序工作，而不是想到哪做哪：

1. 阅读规格文件与当前 `orangecoding-agent` 相关模块，确认现有工作流入口、持久化点与恢复路径。
2. 找到最适合作为 Harness 接入点的上层 orchestrator，而不是从叶子节点开始改。
3. 先定义核心类型与边界：
   - `MissionContract`
   - `HarnessSupervisor`
   - `StepOutcome`
   - `DriftDecision`
   - `ReplanDecision`
   - `HarnessConfig`
4. 再接入最小可运行主流程，让一个长任务可以经历：
   - 建立任务契约
   - 执行一个有界段
   - 产出结果
   - 进入检查点
   - 做出继续/重规划/升级裁决
5. 然后补充配置接入与持久化恢复。
6. 最后补测试、日志、观测与文档。

不要一开始就大面积侵入所有 workflow。先打通一条最小闭环，再扩展覆盖面。

1. `orangecoding-agent` 中当前工作流入口与状态推进逻辑
2. autopilot / boulder 的生命周期边界
3. 会话状态和持久化位置，确认 MissionContract 与检查点最合适的存储点

5. 与用户审批、权限或交互有关的机制，确认升级审阅如何对接

如果你发现现有边界不够清晰，可以做**与本规格直接相关**的小范围结构优化，但不要展开无关重构。

## 八、验收标准

完成后，请逐条自查以下标准是否满足：

1. 是否已经存在独立的 Harness 治理边界，而不是把逻辑塞回 workflow？
2. 是否已经有可持久化的 MissionContract？
3. 是否已经把长任务切成可检查的有界执行段？
4. 是否已经有结构化 `StepOutcome`，而不是只靠自由文本？
5. 是否已经能在检查点做继续 / 重规划 / 升级裁决？
6. 是否已经能阻止明显偏航被静默继续？
7. 是否已经把关键 Harness 策略接入可配置系统？
8. 是否已经有测试证明系统更能保持对齐，而不是仅仅“多了些代码”？

若以上任一问题答案是否定的，请不要声称实现完成。

## 九、实现时禁止出现的反模式

请避免以下错误：

1. **把 Harness 做成 prompt 技巧，而不是代码治理边界**
2. **让 workflow 继续自己判断“是否偏航”，Harness 只做日志**
3. **只记录事件，不做实际门控**
4. **把所有错误统一吞并成一个模糊失败状态**
5. **为了快速通过编译而引入大量 `as any` 风格逃逸、临时 hack 或静默 fallback**
6. **做大规模无关重构，稀释本轮目标**
7. **没有测试就宣称长任务对齐能力已经增强**

如果遇到模糊处，优先选择**更保守、更可验证、更符合规格边界**的实现方式。
