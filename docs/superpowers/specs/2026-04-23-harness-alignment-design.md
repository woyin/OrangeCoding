# Harness Engineering 对齐型长任务控制设计

## 背景

OrangeCoding 已经具备工作流编排、会话连续性、工具权限、多 Agent 协作和终端交互能力，但对于“长时间自治执行时如何持续对准原始任务目标”仍缺少一个独立、可调、可观测的治理层。现有工作流可以负责执行策略，却不应该同时承担“当前动作是否仍然服务于原始目标”的最终裁决职责。

本设计的目标，是引入一个独立的 Harness 层，用来治理长任务中的目标对齐、阶段性复核、重大计划变更审批与配置化调优，使系统在保持较高自治度的前提下，减少长时间运行后逐步偏题、追逐支线问题或在未被察觉的情况下改写任务目标的风险。

## 目标

1. 为长任务提供独立于执行工作流的对齐治理层。
2. 将“是否仍在完成原任务”从工作流内部判断中抽离。
3. 在高自治执行与关键节点人工审阅之间建立清晰边界。
4. 将关键 Harness 策略做成显式配置项，便于持续调优。
5. 为偏航、重规划、升级审批等决策提供明确的可观测依据。

## 非目标

1. 本阶段不重写 autopilot、boulder 或其他既有工作流。
2. 本阶段不追求一次性补齐所有 Claude Code、Codex、Opencode 级别功能。
3. 不将所有执行错误都上升为 Harness 问题；基础设施类失败仍留在执行域处理。
4. 首版不引入过重的全量事件评分体系或复杂策略语言。

## 架构概览

首版在 `orangecoding-agent` 内新增独立的 Harness 子系统，作为长任务执行的顶层治理者。既有工作流（如 autopilot、boulder）继续负责策略执行、工具调用与任务推进；Harness 只负责判定这些动作是否仍然服务于原始任务。

这条边界必须保持清晰：

- **工作流负责“怎么做”**
- **Harness 负责“现在做的事是否仍然还是对的事”**

整个 Harness 围绕一个持久化的 `MissionContract` 运转。该契约记录原始目标、成功标准、已接受计划摘要、禁止偏离的边界、重大计划变更定义、审阅门控策略，以及用户可调的 Harness 配置。执行过程中的每次关键推进都必须围绕该契约进行检查，而不能只依赖工作流内部的即时判断。

## 核心组件

### 1. HarnessSupervisor

长任务的总控入口。负责初始化任务契约、启动执行段、在检查点处触发对齐判定，并决定继续执行、受控重规划或升级给用户。

### 2. MissionContract

长任务的正式任务契约，至少包含：

- 原始任务目标
- 明确成功标准
- 当前已接受计划摘要
- 禁止追逐的旁支问题类型
- 重大计划变更判定规则
- 审阅门控策略
- Harness 配置快照

`MissionContract` 必须可持久化，以支持中断恢复后继续沿用同一任务定义，而不是在恢复时重新解释目标。

### 3. ProgressTracker

在每个执行段结束后记录结构化结果，包括：

- 已完成的推进
- 涉及的文件或资源
- 做出的关键决策
- 当前策略的理由
- 当前阻塞点

它不只记录“发生了什么”，还要记录“为什么当前动作仍被认为有助于目标完成”。

### 4. DriftDetector

偏航识别器。它根据最新 `StepOutcome` 与 `MissionContract` 对比，判断当前执行是否：

- 仍然对齐主任务
- 对主任务有帮助，但已触发需要显式重规划的条件
- 已明显偏离原始目标

首版重点防止的偏航类型是：**失去原始目标，转而追逐看似有价值但并非成功标准一部分的支线问题**。

### 5. ReplanGate

对“可继续 / 需重规划 / 必须升级”做最终裁决。它不负责生成新计划，而负责约束计划变化是否允许在自治模式下发生。

### 6. HarnessConfig

将关键治理策略显式暴露为用户可调配置，而不是散落在 prompt 或代码常量中。该配置应接入现有 OrangeCoding 配置体系统一管理。

## 运行流程

Harness 采用**分段执行**，而不是无限连续运行：

1. 根据用户请求创建 `MissionContract`。
2. 基于当前契约生成初始执行计划。
3. 允许工作流执行一个**有界执行段**，该执行段可包含多个内部步骤。
4. 执行段结束后，工作流返回结构化 `StepOutcome`。
5. Harness 在检查点对本段结果进行对齐判定。
6. 根据判定结果进入三种分支之一：
   - **继续执行**：当前推进仍然服务于目标。
   - **受控重规划**：当前推进仍有价值，但已触发重大方向变化。
   - **升级审阅**：已出现明显偏航，或无法证明当前策略仍对任务有帮助。

整条控制回路为：

**计划 → 有界执行段 → 检查点 → 对齐裁决 → 继续 / 重规划 / 升级**

该回路是首版防止长任务偏题的核心机制。

## 配置模型

首版只暴露少量但关键的显式策略项，避免配置面过大。建议至少包括：

- 检查点频率或执行段长度
- “重大计划变更”的判定阈值
- 偏航检测敏感度
- 置信度下降时的升级策略
- 单次自治推进的最大段长度
- 在重试、重规划、请求用户确认之间的偏好策略

配置设计的目标不是“功能最大化”，而是“让用户清楚系统为何这样决策，并能稳定调节其行为”。

## 错误处理与升级策略

首版必须明确区分两类问题：

### 执行域错误

例如工具失败、网络错误、模型调用失败、外部依赖异常等。这些问题仍由工作流负责处理，可按既有策略重试、降级或阻塞。

### 对齐域错误

当工作流无法证明当前动作仍然服务于 `MissionContract`，或 `DriftDetector` 识别出明显偏航时，应将其视为 Harness 级事件。此时系统必须：

1. 停止继续自治推进；
2. 保留最近一次已接受检查点；
3. 给出具体偏航或升级原因；
4. 进入用户审阅或受控重规划流程。

**禁止静默放行偏航事件。** 一旦检测到偏航，系统不能把它伪装成正常推进。

## 与现有系统的集成原则

为降低首版落地风险，本设计采用“外层治理、内层复用”的集成方式：

1. Harness 作为 `orangecoding-agent` 内的新模块实现，而不是新增独立进程。
2. autopilot、boulder 等现有工作流继续保留执行职责。
3. 工作流向 Harness 输出标准化 `StepOutcome`，而不是让 Harness 直接理解工作流内部细节。
4. 长任务恢复时优先恢复 `MissionContract`、最近已接受检查点与 Harness 配置快照，确保恢复后仍沿用同一治理边界。

这种方式允许系统先把“治理能力”接入，再逐步推动各工作流适配新的监督接口。

## 测试与验证

首版验证必须覆盖三个层级：

### 单元测试

验证检查点策略、偏航分类、重规划门控与配置解析行为是否符合预期。

### 场景测试

模拟长时间运行中的典型偏题诱因，例如：

- 遇到有趣但不属于目标范围的 bug
- 在同一失败路径上反复尝试却不改变策略
- 为了“看起来更完整”而额外扩展任务边界

这些场景必须证明系统会把执行拉回主目标，或升级请求，而不是默默偏航。

### 不变式测试

验证以下关键性质：

- 自治推进只能从已接受检查点继续
- 重大计划变更不能绕过审阅门控
- 检测出的偏航不能被静默降级为成功
- 中断恢复后仍使用同一任务契约，而不是重新解释目标

## 首个里程碑验收标准

首个里程碑不以“完成更多任务”作为主要标准，而以“更稳定地保持任务对齐”作为主要标准。达到以下结果即可视为达成：

1. 长任务执行时存在显式检查点与对齐裁决。
2. 重大计划变更会触发门控，而不是在自治模式下悄然发生。
3. 明显支线任务会被识别为偏航并中止自治推进。
4. 用户可以通过配置调节核心 Harness 策略。
5. 用户能够查看系统为何继续、重规划或请求审阅。

## 结论

该方案通过在现有工作流之上增加独立 Harness 治理层，优先解决 OrangeCoding 在长任务自治过程中的“目标保持”问题。它不试图首版解决全部自主工程能力，而是先把最关键的控制边界、复核回路、偏航判定与配置能力建立起来，为后续持续调优 Harness Engineering 能力打下可演进基础。
